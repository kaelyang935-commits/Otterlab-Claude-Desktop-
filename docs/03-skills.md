---

## 1. 技能指令是什麼？

**技能（Skill）= 一套用提示詞寫好的工作流程。**

當你下一個內建指令（如 `/clear`），程式直接執行一個動作；
但當你下一個技能指令（如 `/code-review`），Claude 會讀取一份「詳細步驟說明書」，然後**自己動用各種工具**（讀檔、跑指令、搜尋…）去完成一整套任務。

> 🧠 **比喻：**
> - 內建指令像「電燈開關」——按下去就亮。
> - 技能指令像「呼叫一位專家」——你說「幫我審查程式碼」，他會自己決定要看哪些檔、跑哪些檢查。

技能可以是 **內建（bundled）** 的，也可以由 **外掛** 或 **你自己**（[第 4 課](04-custom-commands.md)）提供。

---

## 2. 內建常用技能一覽

| 指令 | 做什麼 |
|---|---|
| `/code-review` | 審查目前的程式碼變更，找 bug 與可簡化處 |
| `/simplify` | 只做整理（重用、簡化、效率），不抓 bug |
| `/debug` | 開啟除錯記錄並協助排查問題 |
| `/batch` | 大規模平行改動：拆成多個單元、各派一個背景代理處理 |
| `/loop` | 在階段開著時，重複執行某個提示 |
| `/deep-research` | 大量網搜、交叉查證、產出附引用的研究報告 |
| `/claude-api` | 載入 Claude API 參考資料，協助你寫／升級串接程式 |
| `/run` | 啟動並操作你的 App，確認變更真的有效 |
| `/verify` | 建置＋執行＋觀察，確認變更正確 |
| `/fewer-permission-prompts` | 掃描歷史，幫你減少權限詢問 |

> 用 `/skills` 可列出目前裝置上所有可用技能。

## 3. 看懂技能的參數（以 `/code-review` 為例）

技能常有「等級」與「旗標」可調：

```
/code-review high --fix
            └┬─┘ └─┬─┘
          審查強度  審查後直接套用修正
```

`/code-review` 的審查強度由低到高：

| 等級 | 特性 |
|---|---|
| `low` / `medium` | 數量少、信心高的發現 |
| `high` → `max` | 涵蓋更廣，可能含不確定的發現 |
| `ultra` | 雲端多代理深度審查（最深入） |

常用旗標：

| 旗標 | 作用 |
|---|---|
| `--fix` | 審查後把建議直接套用到工作目錄 |
| `--comment` | 把發現以行內留言貼到 GitHub PR |

**範例：**
```
/code-review            ← 預設強度審查目前變更
/code-review high --fix ← 高強度審查並自動修正
/code-review ultra      ← 雲端深度審查（即原 /ultrareview）
```

---

## 4. 技能 vs 自訂指令的關係

你在 [第 4 課](04-custom-commands.md) 會學到：**「自訂指令」其實就是「自己寫的技能」**。
內建技能跟你自製的技能，呼叫方式完全一樣（都用 `/名稱`），只是來源不同。

| 來源 | 呼叫方式 | 範例 |
|---|---|---|
| 內建 | `/名稱` | `/code-review` |
| 你的專案／個人 | `/名稱` | `/commit`（你寫的） |
| 外掛 | `/外掛名:名稱` | `/my-plugin:deploy` |

## 📝 課堂練習

1. **判斷題：** `/clear` 和 `/deep-research`，哪一個是「技能」？你怎麼判斷？
2. **讀參數：** `/code-review max --comment` 這行會做什麼？拆解每個部分。
3. **實作（需有程式碼專案）：** 隨意改一個檔案，執行 `/code-review`，觀察它讀了哪些檔、提出哪些建議。
4. **探索：** 執行 `/skills`，數數看你的環境裡有多少技能可用。

## 🔑 重點回顧

- 技能 = 用提示詞驅動的「一整套工作流程」，Claude 會自己用工具完成。
- 內建技能：`/code-review`、`/deep-research`、`/run`、`/verify`…
- 等級（low→ultra）控制深度，旗標（`--fix`、`--comment`）控制行為。
